Q1: End-to-End Real-World User Flow with All Edge Cases


The full flow has 7 phases. Here is what actually happens at each stage, and what breaks it:

Phase 1 — Ingest: Data streams in from 4 source types every 60–300 seconds. The first edge case hits here: what if a source feed goes stale or returns null? You need a data freshness timestamp on every edge weight. If weather data is >30 minutes old, the node confidence score drops automatically, and the alert carries a "⚠ stale data" flag.

Phase 2 — Detect: The stress scoring model runs continuously. Critical edge case: false positives. A port showing 90% congestion during a scheduled maintenance window should not trigger a P1 alert. Fix: a calendar layer of known maintenance windows that suppresses alerts or downgrades severity.

Phase 3 — Cascade BFS: The decay factor (0.6^n) is essential here — without it every downstream node turns red. Edge case: circular routes in the graph (A→B→C→A) can cause infinite BFS loops. Fix: visited-node set, standard BFS protection.

Phase 4 — Reroute: Dijkstra runs over the weighted graph. Critical edge case: no valid reroute exists. If all alternative paths are also disrupted (a common scenario during large weather events), the system must surface "no viable alternative — escalate to procurement" rather than silently returning an empty result.

Phase 5 — Alert: LLM brief generation. Edge case: Claude API timeout or rate limit. You need a deterministic fallback brief templated from the structured data, firing in under 200ms if the API call exceeds 1.5 seconds.

Phase 6 — Autonomy gate: The most important design decision in the whole system. The threshold logic should be: auto-execute if (reroute_cost_delta < ₹50,000) AND (shipment_criticality == "high") AND (human_override_toggle == "on"). Edge case: manager is unavailable for manual approval on a time-sensitive shipment. Add an escalation timer — if no action within 15 minutes, the system auto-escalates to the next person in the approval chain.

Phase 7 — Learn: Every prediction outcome (was the disruption real? did the reroute save the projected time?) feeds back to recalibrate the stress scoring model. In a hackathon you mock this — but describe it clearly to judges because it's what makes this defensible as a long-term product.

Q2: What If There Is No Network in the Area?
This is actually a well-studied problem in logistics and the answer has three layers.

The vehicle/driver layer (offline-first): The driver's device should operate on a local-first model. Before a vehicle departs, the ChainPulse mobile app (or onboard unit) downloads the full route graph, waypoint data, and pre-computed alternative routes for the day. This is called an offline bundle — typically 5–15MB per route, easily cached. If connectivity drops mid-route, the driver continues on the last-synced route. No decision is lost because decisions were already pushed down to the device before departure.

The operations center layer (intermittent): The central system requires internet to ingest live feeds, but disruptions don't require real-time response in every scenario. A 10–15 minute sync gap is acceptable for most land freight. The system should use a message queue (in production: Kafka; in your demo: an in-memory queue) so updates accumulate and replay when connectivity resumes. The critical alert — the one that triggers rerouting — only needs to reach the driver once.

The edge case that actually kills you: Connectivity loss during an active rerouting command. If the system has auto-executed a reroute and the execution confirmation never arrives (carrier API didn't acknowledge), you need idempotent retries with exponential backoff. The reroute command carries a unique ID — if it fires twice, the carrier system recognizes the duplicate and ignores the second.

For your hackathon demo: You don't need to build any of this. But when a judge asks "what happens offline?", you say: "The system uses an offline-first architecture. Route decisions are pre-computed and pushed to the driver device before departure. The central system uses a message queue that buffers events during connectivity gaps and replays on reconnect. For the demo we're operating in a connected environment, but the architecture is designed for intermittent connectivity."

Q5: Is Simulation Necessary — Shouldn't AI Just Decide?
This is the sharpest question in the list and the answer is nuanced.

Simulation IS necessary, but not for the reason you might think. The AI should absolutely compute the optimal reroute — that's the core of the system. But simulation serves a different purpose: it lets the manager understand WHY the AI made that recommendation before approving it. Without simulation, you have a black box that says "go via Colombo" and a manager who has no reason to trust it.

The cascade simulation specifically answers: "if I do nothing, here is exactly what breaks, and here is the financial consequence." That is not the AI deciding — that is the AI showing its work. The decision itself (approve/reject the reroute) remains with the human in most cases because:

Carrier contracts have penalty clauses the AI doesn't know about. Customer relationships have informal priorities the data doesn't capture. Regulatory constraints (cabotage rules, customs) can block technically optimal routes. The manager has context the system doesn't.

The bounded autonomy model is the right architecture: The AI auto-executes decisions within a defined confidence and cost envelope (low-stakes, high-confidence, time-critical). It recommends and waits for human approval on high-cost or low-confidence decisions. It escalates to a human and stops on decisions it cannot model (novel disruption type, no valid alternative).

For your hackathon: Show all three modes. The demo scenario should include one auto-execution (fast, satisfying) and one manual approval (shows the human stays in control). That combination is more impressive than either alone.
